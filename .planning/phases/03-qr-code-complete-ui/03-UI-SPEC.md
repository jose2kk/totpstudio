## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps (`gap-1`), inline icon button padding |
| sm | 8px | Compact spacing between label and control (`space-y-2`) |
| md | 16px | Default element spacing (`space-y-4`), column gap |
| lg | 24px | Section spacing within columns (`space-y-6`) |
| xl | 32px | CardContent internal padding |
| 2xl | 48px | Page-level vertical padding (`py-8` = 32px, `py-12` max) |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Touch targets for ghost icon buttons: minimum 32px (size-8 = 32px, existing `size="icon-sm"` pattern)
- QR code display area: 192px × 192px fixed (see QR Code section)
- Progress bar height: 4px (`h-1`) — established in Phase 2

Source: Phase 2 established patterns in `totp-generator.tsx`. Tailwind v4, no `tailwind.config.js`.

## Color

All colors reference shadcn/ui CSS variables (oklch tokens). No raw hex values — use Tailwind utility classes backed by `globals.css` custom properties.

| Role | CSS Variable | Tailwind Class | Usage |
|------|-------------|----------------|-------|
| Dominant (60%) | `--background` | `bg-background` | Page background, outside card |
| Secondary (30%) | `--card` | `bg-card` | Card surface (single Card wraps all content) |
| Primary interactive | `--primary` | `bg-primary` / `text-primary-foreground` | No primary buttons in this phase |
| Muted surface | `--muted` | `bg-muted` | Progress bar track, input backgrounds |
| Muted text | `--muted-foreground` | `text-muted-foreground` | Placeholder text, countdown seconds, empty code state |
| Border | `--border` | `border-border` | Input borders, card borders, column divider (if used) |
| Destructive | `--destructive` | `text-destructive` / `border-destructive` | Secret validation error text and input border |
| Accent (10%) | see below | see below | Countdown bar colors only |

**Accent reserved for:**
1. Countdown progress bar — green phase: `bg-green-500` (>66% remaining)
2. Countdown progress bar — yellow phase: `bg-yellow-500` (33–66% remaining)
3. Countdown progress bar — red phase: `bg-red-500` (<33% remaining)
4. SHA-256/SHA-512 algorithm warning text: `text-amber-500` (inline, below algorithm toggle)

No other chromatic color is used. All interactive elements (buttons, toggles, inputs) use the achromatic neutral token palette.

Source: Phase 1 D-01/D-02 (zinc/neutral, no custom accent), Phase 2 barColor pattern (`bg-green-500`/`bg-yellow-500`/`bg-red-500`), CONTEXT.md D-07 ("amber/yellow text" for SHA warning).

- Two-column grid: `grid grid-cols-2 gap-4` inside CardContent, below the full-width secret input
- Column divider: subtle `border-l border-border` on the right column (or `divide-x divide-border` on the grid) — Claude's discretion to include if visual separation improves clarity

Source: CONTEXT.md D-01, D-02, D-03.

### Mobile (< `md`, 390px target)

Single-column stacking order (top to bottom):
1. Secret input
2. Algorithm / Digits / Period toggles (SHA warning inline below algorithm if applicable)
3. TOTP code display + copy button
4. Progress bar + countdown
5. Issuer input
6. Account Label input
7. QR code (192×192 SVG, centered)

No horizontal overflow. All elements fit within `px-4` container padding.

## Interaction Contracts

### Issuer / Account Inputs

- Both are optional. QR code renders immediately when a valid secret exists, even if issuer and account are empty.
- URI updates live on every keystroke — no submit button needed.
- No validation errors for these fields. Any string (including empty) is valid.
- Placeholder text communicates intent without requiring input.

### QR Code Display

- Size: 192px × 192px SVG (`size={192}`)
- Error correction: Level M (`level="M"`) — balances scan reliability with density
- Background: transparent (`bgColor="transparent"`) so it matches the card surface in both themes
- Foreground: `currentColor` (`fgColor="currentColor"`) — inherits `--foreground` token, renders correctly in dark and light mode
- When secret is invalid or empty: QR code area shows a muted placeholder — `<div className="size-48 bg-muted rounded-md flex items-center justify-center"><QrCode className="size-8 text-muted-foreground" /></div>` (uses lucide `QrCode` icon as empty state visual)

### SHA Warning

- Trigger: algorithm state is `sha256` or `sha512`
- Placement: immediately below the Algorithm ToggleGroup, above Digits control
- Behavior: mounts/unmounts conditionally — no animation needed (inline context, not a toast)
- Copy: "SHA-256 and SHA-512 are not supported by Google Authenticator or Microsoft Authenticator. Use SHA-1 for broad compatibility."
- Class: `text-amber-500 text-xs mt-1`

### otpauth:// URI Display

- Rendered as: `<p className="font-mono text-xs text-muted-foreground truncate">` inside a flex row with copy button
- Full URI visible on hover via `title` attribute for accessibility
- Copy button: ghost icon-sm, Clipboard/Check feedback pattern (1.5s), identical to existing copy buttons
- When secret is invalid or empty: URI display area hidden (`null` render)

### Copy Feedback Pattern (established, replicated)

All copy buttons: show `<Check className="size-4" />` for 1500ms after successful clipboard write, then revert to `<Clipboard className="size-4" />`. No toast or snackbar needed.

## Accessibility Contract

- QR code SVG: `aria-label="QR code for authenticator app"` + `role="img"`
- URI paragraph: `title={fullUri}` for truncation disclosure
- Algorithm warning: `role="alert"` so screen readers announce it when it appears
- Progress bar: existing `role="progressbar"` + `aria-valuenow`/`aria-valuemin`/`aria-valuemax` pattern (Phase 2, unchanged)
- All copy buttons: `aria-label` on every ghost icon button (existing pattern)
- Column layout: semantic order matches visual order on desktop — no `order` CSS tricks that would break tab flow
